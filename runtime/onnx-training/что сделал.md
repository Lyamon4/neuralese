# Что сделал

## 1. Общий статус

Сделал первый рабочий срез on-prem runtime для обучения ONNX-моделей.

Это еще не финальная production-версия, но уже есть нормальная основа проекта:

- отдельный Python-пакет `onprem_runtime`;
- модульное ядро без привязки к FastAPI;
- API-слой поверх ядра;
- минимальный dashboard;
- Docker deployment package;
- systemd deployment docs;
- GitHub Actions CI;
- понятные ошибки API/UI;
- тесты на основную логику;
- локальное окружение для разработки.

Главная цель этого этапа была собрать рабочий вертикальный срез и разложить проект правильно, чтобы runtime можно было дальше расширять без переписывания API и training core.

## 3. Создал структуру runtime

Добавлен новый пакет:

```text
onprem_runtime/
  core/
  api/
  dashboard/
```

Идея такая:

- `core` содержит независимую runtime-логику;
- `api` отвечает только за HTTP/WebSocket;
- `dashboard` содержит минимальный UI для школы;
- тесты лежат отдельно в `tests/onprem_runtime`.

Такой подход нужен, чтобы training engine можно было потом встроить как в школьный on-prem сервер, так и в основной backend.

## 4. Реализовал core-часть

### `core/events.py`

Добавлен единый формат событий для WebSocket.

Пример события:

```json
{
  "job_id": "job_123",
  "phase": "epoch",
  "data": {
    "epoch": 2,
    "epochs": 5,
    "train_loss": 0.42,
    "val_acc": 0.81
  }
}
```

Это нужно, чтобы клиент и dashboard получали прогресс обучения в одном понятном формате.

### `core/config.py`

Добавлен объект `TrainingConfig`.

Он читает параметры из `manifest.json`:

- имя модели;
- task;
- loss;
- optimizer;
- learning rate;
- epochs;
- batch size;
- trainable parameters;
- input/output names.

Также добавлена базовая валидация:

- `epochs` должен быть больше 0;
- `batch_size` должен быть больше 0;
- `trainable_parameters` не должен быть пустым.

### `core/bundle.py`

Реализован parser для входного `.zip` bundle.

Сейчас он проверяет:

- что файл является zip-архивом;
- что внутри есть `manifest.json`;
- что внутри есть `model.onnx`;
- что внутри есть `data/train.npz` или корректный `dataset_ref`;
- что в `train.npz` есть массивы `x` и `y`;
- что zip не содержит небезопасные пути.

На выходе получается `ExtractedBundle`, с которым дальше будет работать trainer.

Bundle теперь может не содержать dataset напрямую, если в `manifest.json` есть:

```json
{
  "dataset_ref": {
    "type": "public",
    "id": "mnist",
    "version": "1"
  }
}
```

или:

```json
{
  "dataset_ref": {
    "type": "local",
    "id": "school-dataset-42",
    "fingerprint": "sha256:..."
  }
}
```

Это нужно, чтобы public/local datasets не передавались целиком внутри каждого ONNX bundle.

### `core/jobs.py`

Добавлена базовая модель training job.

В job хранится:

- `job_id`;
- имя;
- рабочая директория;
- состояние;
- последние метрики;
- путь к snapshot;
- stop flag;
- очередь событий.

### `core/engine.py`

Добавлен основной embeddable engine.

Он умеет:

- принять готовый bundle;
- создать job;
- запустить trainer в фоне;
- отдавать события подписчикам;
- останавливать job;
- возвращать список jobs;
- возвращать конкретную job.
- не считать job успешной, если trainer не вернул реальный snapshot.

Основной интерфейс:

```python
submit_bundle(...)
submit_extracted(...)
stop(job_id)
get_job(job_id)
list_jobs()
subscribe(job_id)
```

Это главный слой, который потом можно будет использовать без FastAPI.

### `core/snapshot.py`

Добавлен сборщик результата обучения.

Он собирает:

```text
snapshot.zip
  manifest.json
  inference.onnx
  metrics.jsonl
  checkpoint/
```

Checkpoint добавляется только если он есть.

Добавлена строгая проверка:

- `inference.onnx` обязателен;
- `metrics.jsonl` обязателен;
- `checkpoint/` опционален;
- если trained model или metrics отсутствуют, snapshot не создается и job должна перейти в `failed`.

### `core/dataset_compression.py`

Добавлен Python mirror dataset compression logic для incremental sync.

Он повторяет важные части компрессии локальных датасетов:

- выбор `rows_per_block`;
- кодирование числовых колонок в 8/16/32-bit;
- quantization для float;
- UTF-8 text columns с нулевым разделителем;
- image bytes columns;
- adaptive raw/RLE block encoding;
- SHA-256 hashes для каждого block.
- декодирование synced packet обратно в training rows.

Это нужно, чтобы runtime мог сравнивать blocks локального dataset и не передавать весь dataset заново.

После последнего шага этот модуль умеет не только кодировать blocks для тестов совместимости, но и раскладывать cached packet обратно в строки вида:

```python
([feature_0, feature_1], label)
```

То есть локальный dataset после incremental sync теперь можно реально передать в training flow.

### `core/dataset_sync.py`

Добавлен первый рабочий слой incremental sync для локальных датасетов.

Сейчас он умеет:

- хранить blocks dataset в runtime cache;
- сравнивать client `block_hashes` с локальным cache;
- возвращать `need` map, то есть какие blocks нужно дослать;
- принимать binary frames в формате старого backend;
- проверять SHA-256 hash payload;
- обновлять cache только нужными blocks;
- собирать packet вида `{"header": ..., "data": [inputs, outputs]}`;
- возвращать `fingerprint` синхронизированного dataset.
- сохранять synced dataset blocks/fingerprint на диск;
- восстанавливать synced datasets после рестарта runtime.

Это основа для того, чтобы локальный dataset не передавался целиком перед каждым training job.

Для default server app cache хранится здесь:

```text
<NEURALESE_STORAGE_DIR>/dataset_sync
```

В Docker deployment это попадает в:

```text
/data/jobs/dataset_sync
```

### `core/datasets.py`

Добавлен слой `DatasetProvider`.

Он нужен, чтобы trainer не знал, откуда физически пришел dataset. Для trainer это всегда один и тот же объект `TrainingDataset`.

Сейчас есть три provider:

- `UploadedDatasetProvider` - берет `x/y` из `data/train.npz` внутри bundle;
- `NeuralesePublicDatasetProvider` - adapter для публичных датасетов через backend-like engine;
- `NpzPublicDatasetEngine` - читает server-side public datasets из папки с `train.npz`;
- `IncrementalLocalDatasetProvider` - берет локальный dataset из `DatasetSyncCache` по `dataset_id` и `fingerprint`.

Также добавлен `DatasetResolver`, который выбирает нужный provider по `dataset_ref.type`:

```text
uploaded
public
local
```

Это разделение важно для следующего шага: bundle сможет содержать не сам dataset, а ссылку на него.

Для `local` provider теперь подключен реальный decoder synced packet. Он проверяет fingerprint, берет blocks из cache, восстанавливает `x/y` и отдает их trainer как обычный `TrainingDataset`.

### `core/ort_trainer.py`

Добавлен первый реальный training adapter через ONNX Runtime Training.

Сейчас `OrtBundleTrainer` делает полный минимальный training flow:

1. Берет `ExtractedBundle`.
2. Загружает `model.onnx`.
3. Загружает dataset через `DatasetResolver` или напрямую из uploaded `train.npz`.
4. Генерирует ONNX Runtime training artifacts:
   - `training_model.onnx`;
   - `eval_model.onnx`;
   - `optimizer_model.onnx`;
   - стартовый checkpoint.
5. Запускает обучение по epochs и batches.
6. После каждого epoch считает:
   - `train_loss`;
   - `val_loss`;
   - `val_acc`;
   - время epoch;
   - количество batches.
7. Отправляет `epoch` events в `TrainingEngine`.
8. Проверяет stop flag до epoch и внутри batch loop.
9. Пишет `metrics.jsonl`.
10. Экспортирует обученную `inference.onnx`.
11. Сохраняет checkpoint, если ONNX Runtime смог это сделать.
12. Собирает strict `snapshot.zip`.

Важно: это первый поддерживаемый path для ONNX Runtime Training, а не гарантия, что любая произвольная ONNX-модель сразу обучится. Сейчас он покрывает нормальный минимальный сценарий: ONNX-модель с trainable initializers, classification loss и uploaded dataset.

Добавлена предвалидация ONNX bundle до запуска тяжелого ORT training:

- если trainable parameter не найден среди model initializers, ошибка показывает конкретное имя и доступные initializers;
- если `input_name` не совпадает с model inputs, ошибка показывает доступные inputs;
- если `output_names` не совпадают с model outputs, ошибка показывает доступные outputs;
- если classification label выходит за диапазон output classes, ошибка показывает допустимый диапазон.

Это нужно, чтобы плохой bundle не выглядел как внутренняя поломка runtime.

`local_school` включен по умолчанию:

Это нужно для cloud scheduler/load balancing.

### `api/server.py`

Добавлен отдельный слой сборки runtime-приложения.

Он нужен, чтобы запуск API не был захардкожен внутри `app.py`.

Сейчас `build_runtime_app(...)`:

- читает профиль из env, если профиль не передан явно;
- создает storage directory;
- создает `TrainingEngine`;
- подключает dataset sync cache;
- собирает `DatasetResolver` для uploaded/public/local dataset refs;
- подключает server-side public dataset cache через `NpzPublicDatasetEngine`;
- подключает decoder для local synced datasets;
- сохраняет local dataset sync cache внутри `storage_dir/dataset_sync`;
- возвращает готовый FastAPI app.

Пример локального запуска:

```bash
uvicorn onprem_runtime.api.server:app --host 127.0.0.1 --port 8010
```

Пример запуска как cloud worker node:

```bash
NEURALESE_RUNTIME_MODE=cloud_node \
NEURALESE_STORAGE_DIR=.neuralese_cloud_node \
NEURALESE_MAX_PARALLEL_JOBS=4 \
uvicorn onprem_runtime.api.server:app --host 127.0.0.1 --port 8010
```

В `cloud_node` dashboard и direct upload выключены по умолчанию, чтобы такой runtime можно было ставить за scheduler/load balancer.

Default server app теперь использует `OrtBundleTrainer`, а не заглушку. То есть запуск через:

```bash
uvicorn onprem_runtime.api.server:app --host 127.0.0.1 --port 8010
```

поднимает runtime с реальным training backend.

Также default server app теперь умеет полный local dataset flow:

```text
WS /ws/datasets/sync
  -> cache blocks
  -> fingerprint
  -> bundle dataset_ref.type=local
  -> ONNX Runtime Training
  -> snapshot.zip
```

## 10. Добавил cleanup старых jobs/snapshots

Добавлен cleanup в `TrainingEngine`.

Он удаляет:

- terminal jobs в состояниях `completed`, `failed`, `stopped`, если они старше retention;
- orphan workspaces из `storage_dir/jobs`, которые остались после рестарта runtime и тоже старше retention.

Running/queued jobs cleanup не удаляет.

Добавлен API endpoint:

```text
POST /api/jobs/cleanup?max_age_seconds=604800
```

Пример:

```bash
curl -X POST "http://127.0.0.1:8010/api/jobs/cleanup?max_age_seconds=604800"
```

Ответ:

```json
{
  "removed_jobs": ["job_..."],
  "removed_orphan_workspaces": ["job_..."],
  "retention_seconds": 604800
}
```

## 11. Добавил auth token для API

Добавлена optional token-защита.

Если token не задан, runtime работает как раньше. Это удобно для локального demo.

Если задан `NEURALESE_AUTH_TOKEN` или CLI-флаг `--auth-token`, protected HTTP routes требуют:

```text
Authorization: Bearer <token>
```

или:

```text
X-Neuralese-Token: <token>
```

Protected WebSocket routes принимают token через query:

```text
?token=<token>
```

Защищены:

- upload/list jobs;
- stop job;
- snapshot download;
- cleanup;
- stats/capacity;
- dataset catalog;
- job WebSocket;
- dataset sync WebSocket.

`GET /api/health` и static dashboard оставлены публичными, чтобы node можно было проверять health-check'ами и открывать UI.

Dashboard получил поле `API token`. Token хранится только в browser local storage и автоматически подставляется в upload/stop/download/API requests.

CLI запуск с token:

```bash
python -m onprem_runtime --mode local_school --host 127.0.0.1 --port 8010 --auth-token school-secret
```

Smoke test тоже умеет проверять защищенный runtime:

```bash
python onprem_runtime/deployment/smoke_test.py --launcher local --port 8130 --auth-token school-secret
```

Auth smoke реально прогнан: bundle upload прошел с Bearer token, job WebSocket прошел с `?token=...`, snapshot download прошел с Bearer token.
